## Configuration

Runtime settings now live in `project_config.json`.

For user-specific overrides, create `project_config.local.json`. That file is ignored by Git and can override paths such as the baseline IDF, EPW, EnergyPlus install, SLURM account, and partition without editing tracked defaults.

The recommended pattern is:

```bash
cp project_config.local.example.json project_config.local.json
```

The IDF and EPW filenames are no longer hardcoded in the scripts. They come from `inputs.idf_path` and `inputs.epw_path` in the config. If `inputs.epjson_cache_path` is omitted, it is derived automatically from `inputs.idf_path`.

The default study output directory is `results/`. Override `project.study_dir` if you want a different location.

## Running the Study

### Option A: Run All Phases Automatically (Recommended)

The orchestrator submits all three phases with SLURM dependencies:

```bash
cd ~/eplus_project

# Default: 1000 sims, auto-tuned batch size for max feasible parallelism
bash run_study.sh

# Custom: 100,000 sims, auto-tuned batch size under array/quota/time constraints
bash run_study.sh 100000

# Override batch size manually if needed
bash run_study.sh 100000 25

# Print the launch plan only, without submitting jobs
PLAN_ONLY=1 bash run_study.sh 100000

# Print the launch plan plus wall-clock scenarios from the latest timing summary
bash plan_study.sh 100000

# Cap concurrency manually if you ever want to, otherwise leave it uncapped
bash run_study.sh 100000 auto 200
```

If `results/` already contains outputs from a previous run,
`run_study.sh` now prints the paths that will be erased and pauses for an
interactive confirmation. Type `y` to continue with a fresh run. Any other
input stops the launcher without submitting jobs.

All study paths, EnergyPlus paths, and SLURM submission settings come from
`project_config.json` plus any local overrides.

Unless noted otherwise, the concrete output examples below assume the default
`project.study_dir` value of `results`.

**How it works:**
1. **Phase 1** — Generates LHS parameter matrix (`lhs_parameters.csv`), zone mapping, and caches the baseline epJSON. Fast single job (~1 min).
2. **Phase 2** — epJSON-only array job where each task generates `in.epJSON` in node-local `$TMPDIR`, simulates with EnergyPlus, parses results, and writes batch outputs. `run_study.sh` auto-tunes `BATCH_SIZE` to maximize task-level parallelism while respecting Vera's SLURM array limit, your home inode quota, and the worker timelimit.
3. **Phase 3** — Merges batch outputs into `study_results.csv` and `hourly_heating.parquet`, adds `dashboard.duckdb` by default, then deletes batch files.

Worker timing is enabled by default. Each worker also writes per-simulation and per-batch timing CSVs into `batch_results/`, and Phase 3 consolidates them into `worker_timings.csv`, `worker_batch_timings.csv`, and `worker_timing_summary.csv`. Set `ENABLE_TIMING=0` if you want to disable this instrumentation. Phase 2 is now epJSON-only.

Dashboard bundle details:
- `dashboard.duckdb` is additive. It does not replace the existing CSV/Parquet outputs.
- It contains normalized tables and views for downstream dashboarding: `manifest`, `runs`, `run_stats`, `calendar`, `zone_catalog`, `parameter_catalog`, `batch_stats`, `study_stats`, `hourly_sparse`, plus `run_dashboard` and `hourly_dashboard` views.
- `hourly_sparse` stores only non-zero zone-hour observations using `(sim_id, hour_index, zone_id, value)` instead of repeating dense zero values.
- Set `EXPORT_DASHBOARD=0` if you want to skip the DuckDB export.
- Set `EXPORT_DASHBOARD_ONLY=1` to build or rebuild `dashboard.duckdb` from existing consolidated outputs without rerunning the CSV/Parquet merge.

Auto-batching details:
- The launcher queries `MaxArraySize` from SLURM.
- It queries `C3SE_quota` for your home file quota and keeps a safety reserve.
- It deletes old log files before submission and discounts them when planning available file headroom.
- It assumes a per-simulation wall time and refuses batch sizes that are too large for the worker time limit.
- By default it leaves the array uncapped so the scheduler can run as many workers as your fair-share and available hardware allow.

Planning command:
- `bash plan_study.sh 100000` prints the same derived batching plan plus worker-phase wall-clock scenarios using the latest `results/worker_timing_summary.csv` if available.
- If no timing summary exists yet, it falls back to the worker time assumption from `run_study.sh`.
- `PLAN_ONLY=1 PLAN_OUTPUT=kv bash run_study.sh 100000` prints the same plan in machine-readable `key=value` form for scripting.

### Option B: Run Phases Individually

**Important:** Always submit from the project root directory (`~/eplus_project`).

For manual submission, build the `sbatch` options from the config with
`eplus_study.config`, then submit the payloads in `slurm/`.

First inspect the current derived plan. Do not hard-code `100` sims per batch unless the planner actually recommends it:

```bash
cd ~/eplus_project

# Print the current plan for review
PLAN_ONLY=1 bash run_study.sh 100000

# Or print the more detailed planning report
bash plan_study.sh 100000

# Extract the current BATCH_SIZE and array spec for manual submission
PLAN_KV=$(PLAN_ONLY=1 PLAN_OUTPUT=kv bash run_study.sh 100000)
while IFS='=' read -r key value; do
   case "$key" in
      PLAN_BATCH_SIZE|PLAN_ARRAY_SPEC) printf -v "$key" '%s' "$value" ;;
   esac
done <<< "$PLAN_KV"

# Build config-driven sbatch arguments once
mapfile -t GENERATE_ARGS < <(python3 -m eplus_study.config sbatch-args generate)
mapfile -t WORKER_ARGS < <(python3 -m eplus_study.config sbatch-args worker)
mapfile -t CONSOLIDATE_ARGS < <(python3 -m eplus_study.config sbatch-args consolidate)

# Phase 1: Generate LHS matrix + baseline epJSON cache
JOB1=$(sbatch --parsable "${GENERATE_ARGS[@]}" --export=ALL,N_SIM=100000 slurm/generate.sh)

# Phase 2: Run epJSON workers after Phase 1 completes
JOB2=$(sbatch --parsable \
   "${WORKER_ARGS[@]}" \
   --dependency=afterok:${JOB1} \
   --array=${PLAN_ARRAY_SPEC} \
   --export=ALL,BATCH_SIZE=${PLAN_BATCH_SIZE},ENABLE_TIMING=1 \
   slurm/worker.sh)

# Phase 3: Consolidate (after Phase 2 completes)
sbatch "${CONSOLIDATE_ARGS[@]}" --dependency=afterok:${JOB2} slurm/consolidate.sh
```

If quota, worker timing, or SLURM limits change, rerun the planning command and use the new `BATCH_SIZE` / array spec instead of reusing an old example.

To rebuild only the additive dashboard bundle for an already consolidated study:

```bash
cd ~/eplus_project
EXPORT_DASHBOARD_ONLY=1 python3 -m eplus_study.consolidate_outputs
```

## Monitoring

```bash
# One-shot progress check
bash monitor.sh

# Auto-refresh every 5 seconds
bash monitor.sh --watch 5

# First inspect what would be deleted before sanitizing the workspace
bash clean_workspace.sh --dry-run

# Then clean local-only files if you really want to reset the workspace
bash clean_workspace.sh

# The monitor scopes itself to the most recently launched study
# and shows compact phase/job summaries instead of every array task.

# Detailed job info
jobinfo -u $USER

# Check a specific job's timing/status after completion
sacct -j JOB_ID --format=JobID,JobName,Start,End,Elapsed,State -X
```

`run_study.sh` now records the latest submitted job IDs in `.run_state.env`
inside the configured study directory, usually `results/.run_state.env`.
`monitor.sh` uses that state to show a concise phase-oriented view: generate,
workers, and consolidate. It also keeps the worker progress at 100% after Phase
3 deletes `batch_results/`, instead of dropping back to zero.

## Storage & Quotas

Your home directory is on the Cephyr file system:
- **Path:** `/cephyr/users/YOUR_CID/Vera`
- **Space quota:** 30 GiB
- **File (inode) quota:** 60,000 files

Check usage with:
```bash
C3SE_quota
```

Each completed simulation creates ~20 output files (~9 MB), but with the
batched pipeline these are generated in node-local `$TMPDIR` and never written
to shared storage. Each batch writes:
- 2 result files (`summary_*.csv`, `hourly_*.parquet`)
- 2 timing files (`timing_detail_*.csv`, `timing_summary_*.csv`) when `ENABLE_TIMING=1`
- 2 SLURM log files (`worker-*.out`, `worker-*.err`)

That means a timed study uses about 6 files per batch while Phase 2 is active.
For example, the current auto plan for 100k sims is 5,000 batches, which is
about 30,000 files during the worker phase. This is why `run_study.sh` checks
`C3SE_quota` and keeps a safety reserve before submitting.

## Key Rules for Vera

1. **Never run simulations on the login node.** Always use SLURM (`sbatch`).
   The login node is shared and only for editing, submitting jobs, and light tasks.
2. **Submit from the project root.** The payload scripts under `slurm/` expect
   the repository root to be the SLURM submit directory so they can resolve
   the project configuration correctly.
3. **Use `$TMPDIR` for I/O-heavy jobs.** `eplus_study.simulate_batch` generates `in.epJSON`
   and EnergyPlus outputs directly in node-local scratch, then writes only the
   batch summary/hourly/timing artifacts back to shared storage.
4. **Cancel jobs cleanly** with `scancel JOB_ID` if you discover a mistake.

## Output

With the default config, final results are saved to
`results/study_results.csv` with columns:

| Column | Description |
|--------|-------------|
| f_wall, f_roof, f_win | Envelope parameter multipliers |
| setpoint | Heating setpoint (°C) |
| hours | Heating schedule hours |
| inf, vent | Infiltration / ventilation (ACH) |
| misc | Miscellaneous gains (W/m²) |
| shgc | Solar heat gain coefficient |
| s_hours, s_close | Shading schedule parameters |
| cop | Boiler COP |
| SIM_ID | Run number (1–N) |
| kWh_sqm_GROUND | Heating energy for ground floor zones (kWh/m²) |
| kWh_sqm_MIDDLE | Heating energy for middle floor zones (kWh/m²) |
| kWh_sqm_TOP | Heating energy for top floor zones (kWh/m²) |

Hourly time-series data is stored in `results/hourly_heating.parquet`
(compressed with zstd). Phase 3 merges the per-batch Parquet files into this
single file and removes the `batch_results/` directory.

Phase 3 also writes `results/dashboard.duckdb` by default. This is the
recommended upload artifact for downstream dashboarding because it keeps
run-level data, timing data, zone metadata, calendar metadata, and sparse hourly
data in one file without duplicating simulation-level columns on every hour.

Timing data is stored in:
- `results/worker_timings.csv` — one row per simulation with phase timings such as `model_load_sec`, `apply_model_edits_sec`, `write_input_sec`, `energyplus_sec`, and `sim_total_sec`
- `results/worker_batch_timings.csv` — one row per batch with medians/means plus batch-level timings such as `input_load_sec`, `baseline_model_load_sec`, `summary_write_sec`, `hourly_concat_sec`, `hourly_write_sec`, and `batch_total_sec`
- `results/worker_timing_summary.csv` — one-row overview with overall `median_*` and `mean_*` values across the full study

Dashboard tables are stored in `results/dashboard.duckdb`:
- `runs` — one row per simulation with input parameters, summary KPIs, run status, and per-simulation timing fields if available
- `run_stats` — per-simulation derived hourly summaries such as annual totals, peak hour, and non-zero counts
- `calendar` — `hour_index` to datetime mapping shared by all simulations
- `zone_catalog` — compact `zone_id` mapping plus zone metadata from `results/zone_mapping.csv`
- `hourly_sparse` — sparse hourly fact table with only non-zero observations
- `batch_stats`, `study_stats`, `manifest`, `parameter_catalog` — supporting metadata and timing summaries

### Retrieving Data

```python
import pandas as pd
import duckdb

# Load aggregated results
results = pd.read_csv("results/study_results.csv")

# Load hourly time-series data
hourly = pd.read_parquet("results/hourly_heating.parquet")

# Get hourly heating for a specific simulation
run_42 = hourly[hourly["sim_id"] == 42]

# Get a specific zone across all simulations
zone_6330 = hourly[["sim_id", "datetime", "zone_6330"]]

# Filter only heating season (non-zero values)
heating_only = hourly[hourly.filter(like="zone_").sum(axis=1) > 0]

# Load only specific columns (fast, avoids reading full file)
subset = pd.read_parquet("results/hourly_heating.parquet",
                         columns=["sim_id", "datetime", "zone_6330"])

# Join hourly data with parameters
params = pd.read_csv("results/lhs_parameters.csv")
params["sim_id"] = params.index + 1
merged = hourly.merge(params, on="sim_id")

# Query the additive dashboard bundle
con = duckdb.connect("results/dashboard.duckdb", read_only=True)

# One row per simulation with KPIs and timing fields
runs = con.execute("SELECT * FROM run_dashboard LIMIT 5").df()

# Sparse hourly data joined to zone and calendar metadata
hourly_sparse = con.execute(
   "SELECT * FROM hourly_dashboard WHERE sim_id = 42 ORDER BY hour_index, zone_id LIMIT 20"
).df()

# Example: annual totals for successful runs only
totals = con.execute(
   "SELECT sim_id, annual_total_heating_j FROM run_dashboard WHERE run_status = 'ok'"
).df()
```
